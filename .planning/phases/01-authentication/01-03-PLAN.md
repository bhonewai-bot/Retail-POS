  <name>Task 1: Create Better Auth catch-all API route</name>
  <files>src/app/api/auth/[...all]/route.ts</files>
  <action>
Create the directory `src/app/api/auth/[...all]/` and the file `route.ts` inside it.

This is a Next.js App Router catch-all API route that proxies all requests under `/api/auth/*` to Better Auth's handler.

Implementation:
1. Import `{ auth }` from `@/lib/auth` (the auth instance created in Plan 01)
2. Import `{ toNextHandler }` from `better-auth/next-js`
3. Create the handler by calling `toNextHandler(auth)`
4. Export the handler as both named exports `GET` and `POST`:

```typescript
import { auth } from "@/lib/auth";
import { toNextHandler } from "better-auth/next-js";

const handler = toNextHandler(auth);

export { handler as GET, handler as POST };
```

Important: The `[...all]` dynamic segment captures all sub-paths, so `/api/auth/sign-in/email`, `/api/auth/sign-out`, `/api/auth/get-session` all route to this handler.

After creating the file, verify the route compiles by running `npm run build` (or check for TypeScript errors). If `toNextHandler` is not available from `better-auth/next-js`, check the installed better-auth version and try the alternative import `{ authHandler }` from `better-auth/next-js` or `{ handler }` from `better-auth/api`.

  </verify>

<task type="auto">
  <name>Task 2: Create full-page login UI with form, error handling, and role-based redirect</name>
  <files>src/app/login/page.tsx</files>
  <action>
Create `src/app/login/page.tsx` — a full-page login screen per D-07.

This must be a Client Component (`"use client"`) since it uses useState, form handling, and client-side auth calls.

Implementation:

1. Mark the file as `"use client"` at the top.

2. Imports: `useState` from react, `useRouter` from `next/navigation`, `authClient` from `@/lib/auth-client`.

3. Component `LoginPage` with local state:
   - `email` (string, default "")
   - `password` (string, default "")
   - `error` (string, default "")
   - `loading` (boolean, default false)

4. `handleSubmit` async function (called on form submit):
   - `e.preventDefault()`
   - Set loading true, clear error
   - Call `await authClient.signIn.email({ email, password })`
   - If the result has an `error` field (non-null), set error to "Invalid email or password" (per D-09 — no confirmation dialog, fast feedback), set loading false, return
   - On success, call `await authClient.getSession()` to get the session
   - Check `session?.user?.role`: if "manager", `router.push("/admin")`, else `router.push("/pos")` (per D-06: managers access all routes, cashiers go to POS)
   - Wrap in try/catch: on exception, set error to "Something went wrong. Please try again." and set loading false

5. Render: Full-page centered layout using Tailwind CSS. Structure:
   - Outer container: full viewport height, centered flex, background color from theme (bg-gray-50 or similar)
   - Card: max-w-md, white background, rounded corners, shadow, padding
   - Header: "Retail POS" as h1, "Sign in to your account" as p subtitle
   - Form with onSubmit={handleSubmit}:
     - Email input: type="email", id="email", placeholder="Email address", required, full width, border, rounded, padding
     - Password input: type="password", id="password", placeholder="Password", required, full width, border, rounded, padding
     - Submit button: type="submit", disabled={loading}, full width, background blue-600, text white, rounded, padding
     - Button text: "Signing in..." when loading, "Sign in" otherwise
   - Error display: When error is non-empty, show a red-bordered div above the form with the error text

6. Export default `LoginPage`.

Follow project conventions: functional component, default export, Tailwind CSS classes (no inline styles), single quotes for strings.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client to /api/auth/* | Credential submission crosses untrusted boundary |
| Server to Client | Session cookie set on successful auth |
| Login page to redirect | Role determines post-login destination |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | /api/auth/sign-in/email | mitigate | Better Auth rate-limits sign-in to 3 attempts per 10 seconds by default |
| T-03-02 | Information Disclosure | Login error messages | mitigate | Generic "Invalid email or password" message — no user enumeration |
| T-03-03 | Elevation of Privilege | Role-based redirect | mitigate | Redirect is convenience only — real RBAC enforced in middleware (Plan 04) |
| T-03-04 | Session Fixation | Session on login | mitigate | Better Auth regenerates session on login (built-in behavior) |
| T-03-SC | Tampering | npm/better-auth install | mitigate | Already verified in Plan 01; no new installs in this plan |
</threat_model>
