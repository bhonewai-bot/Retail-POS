Create `src/app/admin/layout.tsx` — Client component (`'use client'`) that wraps all admin child pages.

Auth check: in useEffect, call `authClient.getSession()`. If no session data, redirect to `/login`. If session user role is not "manager", redirect to `/pos`. Store session in local state. While loading, render a centered "Loading..." message (same pattern as existing admin page).

Layout structure when authenticated:
- Full-width `<div className="min-h-screen bg-gray-50">` container
- A top navigation bar: `<nav className="bg-white shadow">` containing a flex row with the app name "Retail POS" on the left (linked to `/admin`), and the user name + role badge + logout button on the right
- A `<main className="max-w-6xl mx-auto p-8">` content area that renders `{children}`
- Logout button calls `authClient.signOut()` then `router.push('/login')` (exact same pattern as existing admin page)

The session user info (name, role) is fetched in the layout's useEffect and stored in state. Pass the user info to children is not needed — each child page can call `authClient.getSession()` independently if it needs user details.

Create `src/app/admin/products/page.tsx` — Client component.

State:
- `products`: array of product objects (each includes `category` object or null)
- `pagination`: `{ page, pageSize, totalPages, total }`
- `searchQuery`: string (initially "")
- `loading`: boolean

On mount (useEffect): fetch `GET /api/products?page=1&pageSize=20`. Set products and pagination from response.

Search: a text input at the top of the page with placeholder "Search products by name or SKU...". On change, update `searchQuery` state. Debounce the search with a 300ms setTimeout (clear timeout on each keystroke). When searchQuery is non-empty, fetch `GET /api/products/search?q={searchQuery}` and replace the products list with results. When searchQuery becomes empty again, refetch the paginated list from `GET /api/products?page=1&pageSize=20`. Per SEARCH-02, each product row in the table MUST display the stock level.

Product table: rendered in a white card (`bg-white rounded-lg shadow`). Table headers: Name, SKU, Category, Price, Cost, Stock, Status, Actions. Each row renders one product. Price and cost displayed as `$X.XX`. Category shown as category name or "Uncategorized" if null. Status shown as a colored badge: green "Active" if `isActive` is true, red "Inactive" if false. Stock shown as plain number.

Actions column per row:
- "Edit" button (link): `<Link href={/admin/products/${product.id}/edit}>` styled as a small blue text link
- "Deactivate" button (only if `isActive` is true): a button that sends `DELETE /api/products/${product.id}` on click. On success, update the product in the local products array to set `isActive: false` (optimistic update without refetch).

Pagination controls: rendered below the table. Show "Previous" (disabled when page === 1) and "Next" (disabled when page === totalPages) buttons. Show current page number and total pages: "Page {page} of {totalPages}". Clicking Previous/Next refetches `GET /api/products?page={newPage}&pageSize=20`.

Top right of the page: a "New Product" button (link) pointing to `/admin/products/new`, styled as blue-600 button matching the login page button style.

Empty state: when products array is empty and loading is false, show "No products found" message.

Auth guard: check session on mount. If not authenticated or not manager, redirect to `/login` or `/pos` respectively (same pattern as admin layout, but also applied here as defense-in-depth).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser -> Admin UI | Client-side form data before validation; user input crosses to API |
| Admin UI -> API | Authenticated fetch calls from admin pages to product API endpoints |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-05 | Spoofing | Admin layout auth guard | mitigate | Layout checks authClient.getSession() on mount and redirects to /login if unauthenticated; non-manager role redirects to /pos |
| T-02-06 | Tampering | Product form inputs | mitigate | Client-side validation (name, sku, price required); server-side validation in API provides the authoritative check; no client-side-only enforcement |
| T-02-07 | Information Disclosure | Product list page | accept | Product data (including cost) is visible to managers only; the layout auth guard ensures non-managers cannot access these pages |
| T-02-08 | Denial of Service | Search debounce | mitigate | 300ms debounce on search input prevents rapid-fire API calls; API caps search results at 50 items |
</threat_model>
