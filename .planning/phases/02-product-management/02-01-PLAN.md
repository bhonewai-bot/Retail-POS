Check SKU uniqueness via `prisma.product.findUnique({ where: { sku } })` — return 409 if taken. Create product via `prisma.product.create()` with all provided fields. Return 201 with the created product object (serialize Decimal fields to string). Return 400 with `{ error: string }` for validation failures. Wrap all logic in try/catch, return 500 on unexpected errors.

Create `src/app/api/products/[id]/route.ts` — Next.js dynamic route handler exporting GET, PUT, and DELETE functions.

GET: accept `params.id` from the route. Fetch product via `prisma.product.findUnique({ where: { id: Number(params.id) }, include: { category: true } })`. Return 200 with serialized product object, or 404 with `{ error: "Product not found" }`.

PUT: accept `params.id` and JSON body. Validate: `name` (string, non-empty), `sku` (string, non-empty), `price` (number, > 0). If `sku` is being changed, check uniqueness via `prisma.product.findFirst({ where: { sku, NOT: { id: Number(params.id) } } })` — return 409 if taken. Update via `prisma.product.update({ where: { id: Number(params.id) }, data: { ...validFields } })`. Return 200 with updated product (serialized). Return 400 for validation errors, 404 if product not found.

DELETE: accept `params.id`. Perform soft-delete: `prisma.product.update({ where: { id: Number(params.id) }, data: { isActive: false } })`. Return 200 with `{ message: "Product deactivated", product: updatedProduct }`. Return 404 if product not found.

Create `src/app/api/products/search/route.ts` — Next.js route handler exporting GET function.

GET: accept `q` query parameter (the search term). If `q` is empty or missing, return 400 with `{ error: "Search query required" }`. Search via `prisma.product.findMany({ where: { OR: [{ name: { contains: q, mode: "insensitive" } }, { sku: { contains: q, mode: "insensitive" } }], include: { category: true } }, orderBy: { name: "asc" }, take: 50 })`. Serialize all Decimal fields. Return 200 with `{ products: Product[] }`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> API | Untrusted product data crosses into database via POST/PUT routes |
| API -> Database | Prisma queries must not expose raw SQL or internal IDs unintentionally |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | POST/PUT /api/products | mitigate | Validate all required fields server-side (name, sku, price); reject empty/malformed input with 400 before any DB write |
| T-02-02 | Information Disclosure | GET /api/products | accept | No PII in Product model; cost data visible to authenticated managers only (auth already enforced by middleware) |
| T-02-03 | Denial of Service | GET /api/products/search | mitigate | Cap search results at 50 items via Prisma `take: 50`; pagination on list endpoint limits per-page to max 100 |
| T-02-04 | Elevation of Privilege | PUT/DELETE /api/products/[id] | accept | Role enforcement is handled by middleware from Phase 1; API routes do not add separate auth checks (middleware guards all /admin/* and /api/* routes) |
</threat_model>
