<objective>
Fix race conditions in the orders API and inventory adjustments API by moving stock checks inside transactions with pessimistic row locking (SELECT ... FOR UPDATE). Ensure the POS terminal displays clear error messages when a transaction fails due to stock unavailability.

Purpose: Prevent overselling under concurrent checkout scenarios — the core requirement for INV-04. Two cashiers selling the same last item must result in exactly one successful sale.

Output: Updated orders and inventory adjustments API routes with proper locking, plus improved error handling in the POS terminal.
</objective>

    2. Lines 71-110: `prisma.$transaction` creates order, items, and decrements stock (INSIDE transaction)

    The fix: Move the stock check INSIDE the transaction with pessimistic locking.

    **Step 1 — Keep validation outside transaction** (lines 8-43): Input validation (items array, individual items, totals, paymentMethod) stays outside — no DB access needed.

    **Step 2 — Remove the separate stock-check block** (lines 44-62): Delete the entire block that does `product.findMany` + loop checking `product.stock < item.quantity`. This check moves into the transaction.

    **Step 3 — Restructure the transaction** (lines 71-110): Inside `prisma.$transaction(async (tx) => { ... })`:

    a. First, lock all product rows with `SELECT ... FOR UPDATE` using `$queryRaw`:
       ```typescript
       const lockedProducts = await tx.$queryRaw<{ id: number; stock: number; name: string }[]>`
         SELECT id, stock, name FROM "Product"
         WHERE id IN (${Prisma.join(productIds)})
         FOR UPDATE
       ```
       Import `Prisma` from `@/generated/prisma` at the top of the file (add `import { Prisma } from '@/generated/prisma';`).

    b. Build a map of locked products for stock validation:
       ```typescript
       const lockedMap = new Map(lockedProducts.map((p) => [p.id, p]));
       ```

    c. Check stock for each item using the locked data. If any product is not found, throw an error with a descriptive message. If stock is insufficient, throw an error with the product name. Use `throw new Error(...)` — the transaction will catch it and roll back.

    d. Create the order and order items (same as current lines 73-93).

    e. Decrement stock using the locked data — keep the existing `tx.product.update` with `stock: { decrement: item.quantity }` pattern (lines 96-100).

    **Step 4 — Improve error handling in the catch block** (lines 113-116): Distinguish between stock-related errors and other errors. When the error message contains "Insufficient stock" or "Product not found", return a 400 with the specific error message. For other errors, return a 500 as before.

    **Step 5 — Improve POS terminal error display**: In `src/app/pos/page.tsx`, the `PaymentDialog` component's `handlePay` function (line 453) already shows `toast.error('Payment failed', { description: errorData.error })`. The error message from the API now includes the product name (e.g., "Insufficient stock for Widget X"), so the toast will display it. No change needed in the POS terminal for this — the existing error handling is sufficient.

    **Important notes:**
    - Do NOT add a version column to the Product model — pessimistic locking with `FOR UPDATE` is simpler and sufficient for this scale (~200 products, single store)
    - Do NOT use optimistic concurrency — it would require retry logic and is overkill here
    - The `SELECT ... FOR UPDATE` pattern locks rows until the transaction commits or rolls back, preventing concurrent reads of the same rows
    - `Prisma.join` is used to safely interpolate the product IDs array into the raw query
